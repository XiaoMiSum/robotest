## 1. Git 分支模型

```
main          ──────────────────────────────────────
                  ↑ (PR)        ↑ (PR)
develop       ─────●────────────●───────────────────
                 ↗              ↗
feature/*     ──●──●──        ──●
```

| 分支          | 用途     | 来源             | 合并目标           |
| ----------- | ------ | -------------- | -------------- |
| `main`      | 生产就绪代码 | —              | —              |
| `develop`   | 开发主分支  | main           | main           |
| `feature/*` | 功能开发   | develop        | develop        |
| `fix/*`     | Bug 修复 | develop / main | develop / main |
| `release/*` | 发布准备   | develop        | main + develop |
| `hotfix/*`  | 紧急生产修复 | main           | main + develop |

**原则**：

- 禁止直接向 `main` / `develop` 推送代码，必须通过 Pull Request。
- `feature` 分支从 `develop` 拉取，完成后提 PR → `develop`。
- PR 标题格式：`[模块] 描述`（如 `[admin] 新增用户批量禁用功能`）。

## 2. 提交规范

采用 **Conventional Commits**：

```
<type>(<scope>): <description>
```

**type 类型**：

| type       | 说明          |
| ---------- | ----------- |
| `feat`     | 新功能         |
| `fix`      | Bug 修复      |
| `refactor` | 重构（非功能非修复）  |
| `style`    | 代码格式（不影响逻辑） |
| `docs`     | 文档变更        |
| `test`     | 测试相关        |
| `chore`    | 构建、CI、依赖等   |
| `perf`     | 性能优化        |

**scope 约定**：

| scope       | 说明      |
| ----------- | ------- |
| `admin`     | 系统管理模块  |
| `workspace` | 空间管理模块  |
| `project`   | 功能测试模块  |
| `shared`    | 跨模块公共变更 |
| `config`    | 配置类变更   |
| `deps`      | 依赖变更    |

**示例**：

```
feat(admin): 新增用户批量禁用功能
fix(project): 修复评审快照树裁剪时空指针异常
refactor(workspace): 抽取成员权限校验逻辑为公共方法
```

### 提交粒度

- 每个提交保持 **原子性**：一个提交只做一件事。
- 禁止混合「新功能 + 无关的重构或格式化」。
- 提交信息描述 **做了什么** 和 **为什么做**，而非怎么做。

## 3. Pull Request 规范

**PR 模板**：

```
## 关联需求/缺陷
[链接到需求规格或 issue]

## 变更类型
- [ ] 新功能
- [ ] Bug 修复
- [ ] 重构
- [ ] 文档
- [ ] 其他

## 变更内容
[简要描述改了什么]

## 影响范围
[哪些模块/接口可能受影响]

## 本地验证
- [ ] 编译通过
- [ ] 单元测试通过
- [ ] 自测通过（API / 页面）
```

**审查者检查清单**：

- [ ] 是否符合编码风格规范
- [ ] 是否有充分的错误处理
- [ ] 是否有空安全处理
- [ ] 是否添加了必要注释
- [ ] 是否包含单元测试
- [ ] API 变更是否更新了文档
