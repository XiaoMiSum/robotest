## 2. 系统总体设计

### 2.1 架构风格

系统采用 **B/S 架构**，**前后端分离**。整体分为三层：

    ┌─────────────────────────────┐
    │        前端（SPA）           │
    │  ┌────────┐  ┌────────────┐ │
    │  │管理端  │  │  业务端    │ │
    │  └────────┘  └────────────┘ │
    └─────────────┬───────────────┘
                  │ HTTP / WS
    ┌─────────────┴───────────────┐
    │       网关层（Gateway）      │
    │  认证、上下文注入、路由转发  │
    └─────────────┬───────────────┘
                  │
    ┌─────────────┴───────────────┐
    │          服务层              │
    │  ┌─────────┐ ┌───────────┐ │
    │  │管理服务 │ │ 业务服务  │ │
    │  └─────────┘ └───────────┘ │
    │  ┌───────────────────────┐ │
    │  │   WebSocket 服务      │ │
    │  └───────────────────────┘ │
    └─────────────┬───────────────┘
                  │
    ┌─────────────┴───────────────┐
    │          数据层              │
    │  ┌──────┐ ┌──────┐ ┌──────┐│
    │  │关系库│ │缓存  │ │文件  ││
    │  └──────┘ └──────┘ └──────┘│
    └─────────────────────────────┘

### 2.2 前后端职责

* **前端**：
  * 管理端：独立路由树，仅拥有系统角色的用户可见。
  * 业务端：根据活跃工作空间动态加载数据，包含工作空间管理、项目管理、脑图编辑器、测试评审、测试计划、缺陷管理界面。
  * 脑图编辑器支持 WebSocket 实时协作。
* **后端**：
  * 提供 RESTful API，分为管理接口（`/api/admin`）和业务接口（`/api/workspace` 和 `/api/project`）。
  * WebSocket 服务负责脑图文档的实时协作消息广播与持久化。
  * 中间件统一处理 JWT 认证、活跃工作空间/项目上下文注入与校验。

### 2.3 技术选型概要

| 组件    | 选型方向                                     |
| ----- | ---------------------------------------- |
| 前端框架  | 现代 SPA 框架（React / Vue / Angular）         |
| 脑图编辑器 | 自研或基于开源库扩展，需支持自定义数据模型                    |
| 后端框架  | 主流 Web 框架（Spring Boot / Express / Gin 等） |
| 数据库   | 关系型数据库（MySQL / PostgreSQL）               |
| 缓存    | Redis                                    |
| 实时通信  | WebSocket（Socket.IO 或原生实现）               |

## 3. 模块划分

### 3.1 模块总览

    软件测试平台
    ├── 系统管理模块
    │   ├── 用户管理
    │   ├── 全局工作空间管理
    │   └── 系统角色与权限管理
    └── 业务功能模块
        ├── 我的工作空间
        ├── 工作空间成员管理
        ├── 项目管理
        └── 功能测试
            ├── 测试用例管理（模块树 + 脑图协作编辑）
            ├── 测试评审管理
            ├── 测试计划管理
            └── 缺陷管理

### 3.2 模块职责简述

* **用户管理**：账号生命周期维护，分配系统角色，管理工作空间归属。
* **全局工作空间管理**：工作空间的创建、解散，全平台成员分配。
* **系统角色与权限管理**：定义系统级角色及其管理端权限，系统管理员可配置。
* **我的工作空间**：查看用户归属的工作空间列表，切换活跃空间，编辑基本信息。
* **工作空间成员管理**：管理当前工作空间内成员，可邀请/移除，并可设置空间管理员。
* **项目管理**：当前工作空间下项目的 CRUD 与归档。
* **测试用例管理**：模块树管理 + 脑图实时协作编辑 + 用例列表。
* **测试评审管理**：发起用例评审，基于快照进行节点标记/评论，支持记录与同步。
* **测试计划管理**：创建计划，基于快照跟踪执行，支持多次执行记录与同步。
* **缺陷管理**：缺陷全生命周期管理，关联用例与计划。

## 4. 核心机制设计

### 4.1 角色与权限模型

#### 4.1.1 角色分类

* **系统角色**：由系统管理员在管理端创建和管理，用于控制管理端（`/api/admin`）功能权限。系统预置“系统管理员”角色，拥有全部管理权限且不可删除。系统角色仅分配管理端权限。
* **业务角色**：内置固定角色，不可由管理员创建或删除。包含：
  * 默认“业务用户”角色（所有进入工作空间的成员自动获得）。
  * “空间管理员”角色，在工作空间内部授予，拥有该工作空间内更高业务权限（如修改空间信息、管理成员、删除项目等）。
* 业务用户的最终权限由业务角色和当前工作空间内角色共同决定。

#### 4.1.2 多角色支持

* 一个用户可以同时拥有多个系统角色和多个业务角色。
* 用户的最终权限是所有角色权限的并集。
* 系统管理员在用户管理中只能分配和调整**系统角色**；业务角色由工作空间上下文自动获得（如空间管理员由系统管理员或现有空间管理员在成员管理中设置）。

#### 4.1.3 工作空间内角色

* 用户-工作空间关联表增加 `workspace_role` 字段，可选值：`member`（普通成员）、`admin`（空间管理员）。
* 系统管理员在全局工作空间管理界面添加成员时，可勾选设为空间管理员；后续空间管理员也可在“工作空间成员管理”中提升或降级成员。
* 空间管理员拥有该工作空间内更高的业务权限（具体权限在业务模块详细设计中定义）。

### 4.2 认证与上下文

* 登录后颁发 JWT（短期访问令牌 + 长期刷新令牌）。
* 网关中间件验证 Token，提取用户 ID、所有角色及合并后的权限列表。
* 管理端请求：仅需验证用户拥有至少一个系统角色，通过后即可访问。
* 业务端请求：必须携带 `X-Active-Workspace`，中间件校验用户是否属于该工作空间；项目内操作另需 `X-Active-Project`。同时根据用户在工作空间内的角色（member/admin）和所拥有的业务角色进行授权判断。
* 角色或权限变更后，强制相关用户的 Token 失效（通过版本号或黑名单机制），要求重新登录。

### 4.3 数据隔离

* 管理端接口不注入工作空间上下文，全局查询。
* 业务接口在数据库查询层自动附加 `workspace_id` 或 `project_id` 过滤条件，防止越权访问。

### 4.4 脑图实时协作

* 用户进入文档后建立 WebSocket 连接，加入文档房间。
* 编辑操作以增量消息发送服务端，校验权限与工作空间归属后持久化并广播。
* 冲突处理采用乐观锁或 CRDT 算法保证最终一致性。

### 4.5 快照机制

* **生成**：评审或计划发起/关联用例时，复制当前项目完整模块树 + 涉及文档的全部脑图节点。
* **存储**：存入专用快照表，与原始数据解耦。
* **裁剪展示**：前端请求时，服务端根据关联节点向上找祖先、向下找子孙，仅返回关联节点树。
* **同步**：手动触发，用原始节点 ID 匹配更新快照；已删除节点标记为失效。

### 4.6 评审与执行记录

* **评审记录**：每次标记/评论生成一条记录，并同步更新快照节点的最后标记字段。
* **执行记录**：每次执行生成一条记录，并同步更新快照节点的最后执行结果字段。
* **历史追溯**：通过记录表按时间排序获取完整历史。

## 5. 数据设计

### 5.1 核心实体关系图

    User ──── UserRole ──── Role (type: system / business)
    User ──── UserWorkspace (workspaceRole: member / admin) ──── Workspace
    Workspace ──── Project ──── TestCaseModule
                     ├── TestPlan ─── PlanModuleSnapshot
                     │                PlanNodeSnapshot
                     │                PlanExecutionRecord
                     ├── TestReview ─── ReviewModuleSnapshot
                     │                  ReviewNodeSnapshot
                     │                  ReviewRecord
                     └── Bug

### 5.2 核心数据对象概要

| 对象                   | 描述                                  |
| -------------------- | ----------------------------------- |
| User                 | 账号、邮箱、密码哈希、状态                       |
| Role                 | 名称、类型（system/business）、权限点列表、是否系统预置 |
| UserRole             | 用户与角色多对多关联                          |
| Workspace            | 名称、描述、状态                            |
| UserWorkspace        | 用户-工作空间关联，含 `workspaceRole` 和加入时间   |
| Project              | 名称、描述、状态、所属工作空间                     |
| TestCaseModule       | 模块树节点（目录/文档）                        |
| TestCaseNode         | 脑图节点（用例/普通），树形结构                    |
| TestPlan             | 计划基本信息                              |
| PlanModuleSnapshot   | 计划-模块快照                             |
| PlanNodeSnapshot     | 计划-节点快照，冗余最后执行结果                    |
| PlanExecutionRecord  | 执行记录                                |
| TestReview           | 评审基本信息（发起人、参与者）                     |
| ReviewModuleSnapshot | 评审-模块快照                             |
| ReviewNodeSnapshot   | 评审-节点快照，冗余最后评审标记                    |
| ReviewRecord         | 评审记录（标记/评论）                         |
| Bug                  | 缺陷信息，状态流转                           |

### 5.3 数据隔离原则

业务数据通过所属项目 → 工作空间间接隔离，查询时强制附加上下文条件。管理端无此限制。

## 6. 接口设计概要

### 6.1 接口风格与约定

* 协议：HTTPS，数据格式 JSON。
* 认证：请求头 `Authorization: Bearer <token>`。
* 接口分类：
  * 管理接口：`/api/admin`，需要系统角色。
  * 工作空间级业务接口：`/api/workspace`，需头 `X-Active-Workspace`。
  * 项目内业务接口：`/api/project`，需头 `X-Active-Project`。
* 上下文传递：工作空间和项目 ID 通过请求头传递，不在 URL 中暴露。
* 通用响应格式：`{ "code": 200, "message": "success", "data": {...} }`

### 6.2 管理端接口概要

* **用户管理**：CRUD，支持多系统角色分配，工作空间归属及空间管理员设置。
* **工作空间管理**：创建、解散、成员管理（含设置空间管理员）。
* **系统角色管理**：CRUD，权限配置（仅系统角色）。
* **权限点树**：获取管理端权限树（按模块分组）。

### 6.3 业务端接口概要

* **我的工作空间**：查看归属列表、切换活跃空间、编辑信息。
* **工作空间成员管理**：邀请、移除、设置空间管理员。
* **项目管理**：CRUD、归档。
* **测试用例**：模块树、脑图节点、用例详情。
* **测试评审**：发起、快照树、评审记录、同步。
* **测试计划**：创建、快照树、执行记录、同步。
* **缺陷管理**：CRUD、状态流转。

详细端点定义参见《API 详细设计文档》。

### 6.4 WebSocket 接口概要

* 端点：`/ws/documents/:docId`
* 认证：连接时通过查询参数传递 Token。
* 消息基于 JSON，支持节点增删改移、布局更新。
* 广播至同一文档的在线用户。

## 7. 部署视图

### 7.1 前后端分离部署（方案 A）

* 前端静态资源部署到 CDN/Nginx，后端服务集群可水平扩展。
* 适用于大规模独立迭代。

### 7.2 前后端合并打包部署（方案 B）

* 前端构建产物嵌入后端服务，统一部署。
* 适用于中小团队快速交付。

## 8. 安全设计

* 全链路 HTTPS / WSS。
* 多角色权限并集计算；角色或权限变更后强制 Token 失效。
* 管理接口仅限拥有系统角色的用户访问。
* 业务接口强制校验工作空间归属及工作空间内角色。
* 防暴力破解、CSRF、XSS。
* WebSocket 连接验证 Token 及工作空间上下文。

***

## 9. 设计约束与原则

* **角色分离**：系统角色仅用于管理端，业务角色和工作空间角色用于业务端，权限边界清晰。
* **快照独立性**：快照表与业务表独立，不建立外键引用。
* **权限扩展性**：权限点树形管理，新增功能追加节点即可。
* **上下文集中管理**：工作空间与项目上下文通过请求头统一注入，路由简洁。
