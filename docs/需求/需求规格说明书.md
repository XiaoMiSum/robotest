## 1. 引言

### 1.1 编写目的

本文档定义软件测试平台的功能需求与非功能需求，面向项目经理、业务分析人员、测试工程师及开发团队，确保各方对平台的行为、规则和边界有一致理解。

### 1.2 范围

平台包含两类用户角色：**系统管理员**和**业务用户**。  

- 系统管理员拥有全局管理权限，可管理用户、工作空间和角色。  
- 业务用户可同时归属多个工作空间，通过切换活跃工作空间在不同工作空间间工作，所有业务操作限定在当前活跃工作空间内。

测试用例以 **模块树（目录/文档）** 组织，每个文档包含一个脑图形式的测试用例集合，支持多人实时协作编辑。  
测试评审和测试计划基于用例的结构化快照进行，支持完整的历史回溯和多次执行/评审记录。

### 1.3 定义与缩写

| 术语         | 定义                                         |
| ---------- | ------------------------------------------ |
| 活跃工作空间     | 业务用户登录后选择的工作空间，所有业务操作默认在该工作空间上下文中执行        |
| 模块         | 项目内测试用例的组织结构，分为**目录**和**文档**两种节点           |
| 目录         | 仅用于分类的模块节点，不包含测试用例                         |
| 文档         | 关联一个脑图的模块节点，包含若干测试用例                       |
| 脑图文档       | 以思维导图形式组织的测试用例集合，由多个节点组成                   |
| 测试用例（用例节点） | 脑图中被标记为“用例”的节点，具有步骤、预期结果等属性                |
| 快照         | 评审或计划创建时对模块树和文档节点的完整复制，用于固化当时的状态           |
| 关联节点       | 用户在创建评审或计划时显式选择的用例节点                       |
| 关联节点树      | 从快照中裁剪出的最小树形结构，包含所有关联节点、它们的祖先模块/文档节点以及后代节点 |
| 同步         | 手动将评审或计划的快照更新为当前最新模块和用例内容的操作               |

## 2. 总体描述

### 2.1 用户角色与特征

| 角色    | 职责概述              | 权限边界                         |
| ----- | ----------------- | ---------------------------- |
| 系统管理员 | 平台全局维护            | 无工作空间归属，不可访问任何业务功能           |
| 业务用户  | 所属工作空间内的测试活动管理与执行 | 可同时归属多个工作空间，所有业务操作限于当前活跃工作空间 |

- 一名用户只能拥有一种主要角色（系统管理员或业务用户）。
- 系统管理员由系统初始化时预设，不可被删除。
- 业务用户至少归属一个工作空间才能使用业务功能。

### 2.2 假设与依赖

- 平台账号由系统管理员统一创建，不存在自主注册功能。
- 平台具备基本的日志与审计能力（本文档不展开）。
- 脑图编辑器的前端技术选型及实时协作实现方案不影响业务需求的定义。

## 3. 功能需求

### 3.1 系统管理模块（仅系统管理员可访问）

系统管理员登录后仅可见系统管理功能，无任何业务模块入口。

#### 3.1.1 用户管理

**功能描述**：管理平台所有账号的生命周期及工作空间归属。  
**功能点**：  

- 创建账号：录入用户名、邮箱、初始密码，选择角色；若为业务用户，可同时指定一个或多个初始工作空间。  
- 编辑账号：修改邮箱、角色、密码重置；调整业务用户的工作空间归属（添加或移除工作空间）。  
- 禁用/启用账号：禁用后该用户无法登录，历史数据保留。支持批量操作。  
- 查询与筛选：可按用户名、邮箱、状态、角色、所属工作空间等条件检索用户。  

**业务规则**：  

- 用户名必须全局唯一，长度3~30字符。  
- 系统管理员不能修改自己的角色或禁用自己。  
- 若移除某业务用户的所有工作空间归属，该用户登录后将无法使用任何业务功能。

#### 3.1.2 全局工作空间管理

**功能描述**：管理平台内所有工作空间的创建、解散及成员构成。  
**功能点**：  

- 创建工作空间：输入工作空间名称和描述，名称全局唯一，长度2~50字符。  
- 解散工作空间：仅当工作空间下没有任何项目（含归档项目）时允许解散；解散后自动解除所有成员的该工作空间归属。  
- 添加成员：选择目标工作空间，从平台业务用户中多选加入。一个用户可同时属于多个工作空间。  
- 移除成员：解除指定用户与目标工作空间的关系，不影响该用户在其他工作空间的归属。  

**业务规则**：  

- 系统管理员不能被添加为任何工作空间成员。

#### 3.1.3 角色与权限管理

**功能描述**：定义平台角色及其功能权限，并将角色授予用户。  
**功能点**：  

- 定义角色：可创建自定义角色，预置“系统管理员”和“业务用户”角色不可删除。  
- 配置权限：以功能模块为单位，通过权限列表勾选可访问的功能点。系统管理员默认拥有所有管理权限且不可更改；业务用户默认拥有所有业务权限，可根据自定义角色进行缩减。  
- 分配角色：为用户指定一个主要角色。  

**业务规则**：  

- 角色名称全局唯一。  
- 权限变更后，已登录用户需重新登录方可生效。

---

### 3.2 业务功能模块（仅业务用户可访问）

业务用户登录后，如果归属多个工作空间，必须选择一个作为活跃工作空间；如果仅归属一个工作空间，则自动激活。界面提供工作空间切换器，切换后所有业务数据立即刷新至新工作空间上下文。

#### 3.2.1 我的工作空间

**功能描述**：查看用户归属的工作空间列表，切换活跃工作空间，并可修改工作空间基本信息。  
**功能点**：  

- 工作空间列表：显示用户归属的所有工作空间，标注当前活跃工作空间。  
- 切换活跃工作空间：点击非活跃工作空间即可切换，全局状态更新。  
- 修改工作空间信息：进入工作空间详情，可编辑工作空间名称和描述（名称需全局唯一）。  

**业务规则**：  

- 若用户仅归属一个工作空间，可自动激活，工作空间切换器可隐藏。  
- 若用户从当前活跃工作空间中被移除，系统应自动切换到剩余工作空间中的第一个；若无剩余工作空间，则业务功能不可用。

#### 3.2.2 工作空间成员管理（当前活跃工作空间）

**功能描述**：管理当前活跃工作空间的成员构成。  
**功能点**：  

- 查看成员列表：显示当前工作空间所有成员及其角色。  
- 邀请成员：从平台其他业务用户中搜索并邀请加入当前工作空间。若对方已是本工作空间成员则提示。  
- 移除成员：解除某成员与当前工作空间的关系，不可移除自己。  

**业务规则**：  

- 所有操作仅作用于当前活跃工作空间，不影响用户在其他工作空间的归属。

#### 3.2.3 项目管理

**功能描述**：管理当前活跃工作空间下的项目。  
**功能点**：  

- 项目列表：展示当前工作空间所有项目，显示名称、状态、起止时间等信息。  
- 创建项目：输入名称（工作空间内唯一）、描述、起止时间，项目自动归属当前工作空间。  
- 编辑项目：可修改名称、描述、时间等信息，但所属工作空间不可变更。  
- 归档/启封：归档后项目数据变为只读；归档前需确保无进行中的测试计划。支持反归档。  

**业务规则**：  

- 切换活跃工作空间后，项目列表自动刷新。  
- 用户进入项目时，系统需验证该项目属于当前活跃工作空间。

#### 3.2.4 功能测试（项目工作区）

进入项目后，所有功能测试活动均在当前选定的项目内进行，当前活跃工作空间上下文不变。

##### 3.2.4.1 测试用例管理

**模块管理**  

- 项目下可创建无限层级的模块树，节点类型分为**目录**（可包含子目录和文档）和**文档**（不可包含子模块，仅关联一个脑图）。  
- 支持新建、重命名、删除、拖拽排序。删除目录时需目录为空；删除文档将同时删除其内部所有用例数据。  

**脑图实时协作编辑**  

- 打开文档后进入脑图编辑器，支持同时有多名用户编辑同一文档，编辑操作实时同步给其他用户。  
- 脑图由任意节点构成，用户可自由创建、移动、删除节点，并可编辑节点文本、调整层级关系。  
- 节点可被标记为**用例节点**，用例节点额外拥有优先级、前置条件、测试步骤、预期结果、标签等属性。  
- 编辑过程中系统应保证数据一致性，多人操作冲突时以合理策略合并。  

**用例列表视图**  

- 提供列表模式查看当前文档中的所有用例节点，支持按标题、优先级筛选，可在列表中编辑用例属性，并与脑图视图双向同步。  

**关联与引用**  

- 在测试计划或测试评审中，用户可以从模块树中选取具体的用例节点进行关联。  
- 关联完成后，系统生成该时刻的模块和文档快照，以保证后续历史查看不受用例修改影响。

##### 3.2.4.2 测试评审管理

**概述**  
测试评审仅针对测试用例，以脑图快照为基础，支持多位参与者对节点进行标记和评论，同一节点可被多次评审，形成评审记录。

**发起评审**  

- 用户从模块树中选择需评审的用例节点（可跨文档多选），指定发起人（默认自己）和若干参与者（必选至少一人），填写标题和描述。  
- 系统根据所选节点，自动生成当时的模块快照和文档节点快照，评审状态初始为“评审中”。

**评审详情展示**  

- 评审详情以**关联节点树**的形式呈现：仅展示与关联节点相关的模块路径和脑图子树，高亮被标记为关联的用例节点，并显示各节点的最新评审标记状态。  

**评审操作**  

- 参与者可点击树上的任意节点，进行**标记**（仅用例节点可标记“通过”或“不通过”）或**添加评论**。  
- 每次提交生成一条评审记录，系统同步更新该节点的最后标记状态（用于树形视图的快速展示）。  
- 所有参与者的操作实时保存，其他参与者可即时看到最新状态。  

**评审记录查看**  

- 对于任意节点，可查看其完整的评审历史，包含历次标记变更和评论。  

**完成评审**  

- 发起人可手动关闭评审，状态变为“已完成”，此后不可再新增或修改评审记录，快照冻结。  

**同步最新用例**  

- 在“评审中”状态下，发起人可执行同步操作。系统根据关联的原始节点ID，用当前最新的模块结构和用例内容更新快照。若某原始节点已被删除，则在快照中标记为“已删除”，其已有评审记录仅可查看。  
- 同步后，保留所有已产生的评审记录。  

**业务规则**：  

- 参与者必须是当前活跃工作空间成员。  
- 发起人不能同时作为参与者。  
- 评审完成后，任何人均只能查看，不能编辑。

##### 3.2.4.3 测试计划管理

**概述**  
测试计划关联测试用例并管理执行过程，基于快照进行，支持多次执行记录，以追踪执行历史。

**创建计划与关联用例**  

- 用户创建计划，输入名称、起止时间、负责人、执行环境、描述等信息。  
- 从模块树中选择需执行的用例节点，系统生成当时的模块和文档快照，并为每个关联的用例节点初始化执行状态为“未执行”。

**计划详情展示**  

- 计划详情以**关联节点树**形式展示，高亮关联的用例节点，并根据最后一次执行结果显示状态图标（通过/失败/阻塞/未执行）。  

**执行跟踪**  

- 负责人或指定的执行人可点击树上的关联用例节点，标记执行结果（通过/失败/阻塞/未执行）并填写备注。  
- 每次提交生成一条执行记录，系统同步更新该节点的最后执行结果。  
- 同一节点可被多次执行，所有历史记录可追溯。  

**执行历史**  

- 对任一节点可查看其完整的执行记录列表，包含每次结果、执行人和备注。  

**计划状态流转**  

- 计划状态按“新建 → 进行中 → 已完成 → 已关闭”流转。  
- 关闭计划时，系统检查是否有未执行用例，若有则提示。关闭后不可再修改执行结果。  

**同步最新用例**  

- 在“进行中”状态下，负责人可同步快照至最新版本，保留所有已产生的执行记录。已删除的原始节点将在快照中标记，其结果只读。  

**业务规则**：  

- 计划负责人和实际执行人必须是当前工作空间成员。  
- 只有被关联的用例节点才可以被标记执行结果。

##### 3.2.4.4 缺陷管理

**功能描述**：管理项目内发现的缺陷，支持完整生命周期跟踪。  
**功能点**：  

- 提交缺陷：填写标题、严重等级（致命/严重/一般/轻微）、优先级（高/中/低）、详细描述、附件，可关联至当前版本的测试用例和测试计划。  
- 缺陷状态流转：新建 → 已指派 → 修复中 → 已修复 → 已验证 → 关闭，可重新打开。  
- 指派处理人：从当前工作空间成员中选择。  
- 操作记录：每次状态变更或信息更新时记录注释，形成操作历史。  
- 统计视图：按状态、严重等级、处理人等维度展示缺陷分布图表。  

**业务规则**：  

- 处理人必须是当前活跃工作空间成员。  
- 关闭缺陷时必须填写关闭说明。

## 4. 非功能需求

### 4.1 性能需求

- 常规页面首屏加载时间不超过2秒。  
- 工作空间切换后数据刷新应在500毫秒内完成。  
- 脑图文档（含200个以上节点）打开时间不超过1秒。  
- 快照生成操作应在500毫秒内完成（对于200节点规模的文档）。  
- 系统应支持至少100名业务用户同时在线操作。

### 4.2 安全性需求

- 用户身份认证应安全可靠，会话超时后需重新登录。  
- 业务操作必须验证用户所属的活跃工作空间，严格隔离不同工作空间的数据。  
- 登录失败次数过多时应临时锁定账户。  
- 应防范常见的Web安全攻击（如跨站脚本、跨站请求伪造）。  
- 实时协作通道也应进行身份验证和工作空间上下文校验。

### 4.3 可用性需求

- 脑图编辑器应提供常用键盘快捷键（如Tab/Enter创建子节点，Delete删除节点）。  
- 工作空间切换器应固定在界面明显位置，当前活跃工作空间状态清晰可见。  
- 在快照展示页面，应明确标识“快照版本”和生成时间。  
- 同步操作应提供二次确认，防止误操作。  
- 执行历史和评审历史应支持按时间排序查看。

### 4.4 可靠性需求

- 系统应保证7×24小时可用，计划内维护时间每月不超过2小时。  
- 数据应每日自动备份，支持恢复，数据丢失不超过24小时。

## 5. 数据需求

### 5.1 核心数据实体及其关系

- **用户**：存储账号信息、角色及状态，与工作空间为多对多关系。  
- **工作空间**：存储工作空间基本信息，是业务资源隔离的基本单位。  
- **项目**：归属于某个工作空间，包含若干模块和缺陷。  
- **测试用例模块**：形成树形结构，节点分为目录和文档。  
- **脑图节点**：归属于某个文档，形成树形结构，节点可标记为用例节点，用例节点包含测试属性。  
- **测试计划**：归属于项目，包含计划信息，与快照关联。  
- **计划快照**：计划创建时生成的模块和节点副本，与执行记录关联。  
- **测试评审**：归属于项目，包含发起人、参与者列表，与快照关联。  
- **评审快照**：评审创建时生成的模块和节点副本，与评审记录关联。  
- **缺陷**：归属于项目，可关联原始用例和测试计划，具有状态流转和操作日志。

### 5.2 数据隔离

所有业务数据（项目、用例、计划、评审、缺陷）均通过工作空间归属进行隔离，确保业务用户只能访问其当前活跃工作空间内的数据。
