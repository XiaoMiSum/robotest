## 1. 引言

### 1.1 编写目的

本文档对软件测试平台中的**功能测试模块**进行详细设计，定义数据结构、接口规范、业务逻辑、前端组件及交互流程，为开发实现提供完整依据。

### 1.2 范围

功能测试模块面向**业务用户**，在已选定的项目内提供测试用例管理、测试评审、测试计划及缺陷管理功能。所有操作限定在当前活跃工作空间和项目上下文中，项目上下文通过请求头 `X-Active-Project` 传递。

- 《软件测试平台需求规格说明书 V2.1》

## 3. 接口详细设计

### 3.1 通用约定

- 项目内接口基础路径：`/api/project`
- 认证：`Authorization: Bearer <token>`
- 业务请求头：必须带 `X-Active-Project`
- 命名风格：camelCase
- 分页：`page`、`pageSize` → `{ records: [], total: number }`
- 通用响应：`{ "code": 200, "message": "success", "data": {} }`

### 3.2 项目工作台接口

#### 3.2.1 获取项目工作台数据

- **路径**：`GET /api/project/dashboard`

- **说明**：返回当前项目的概览统计数据。

- **响应**：
  
  ```json
  {
  "caseCount": 256,
  "activeReviewCount": 3,
  "activePlanCount": 2,
  "openBugCount": 12,
  "recentReviews": [
    { "id": 1, "title": "登录模块评审", "status": "completed", "createdAt": "2026-07-01T10:00:00Z" }
  ],
  "recentPlans": [
    { "id": 1, "name": "功能测试计划", "status": "in_progress", "createdAt": "2026-07-05T09:00:00Z" }
  ]
  }
  ```

### 3.3 测试用例管理接口

#### 3.3.1 获取模块树

- **路径**：`GET /api/project/modules`
- **响应**：直接返回树形数组，每个节点包含 id、name、type、children。

#### 3.3.2 创建模块节点

- **路径**：`POST /api/project/modules`
- **请求体**：`{ "parentId": null, "type": "directory", "name": "新目录" }`
- **校验**：同父节点下名称唯一。
- **文档类型特殊处理**：当 `type=document` 时，事务中创建文档节点并自动创建默认根节点（title 与文档 name 一致，type='normal'）。
- **响应**：创建后的模块节点信息，状态码 201。

#### 3.3.3 更新/删除模块节点

- **路径**：`PUT/DELETE /api/project/modules/:id`
- 更新：可修改 name。
- 删除：目录需为空；文档级联删除所有节点及布局。

#### 3.3.4 获取文档全量节点

- **路径**：`GET /api/project/documents/:docId/nodes`

- **说明**：返回文档的完整脑图结构。node 为文档的根节点对象（嵌套树形结构），layout 为布局信息。根节点是创建文档时自动生成的默认节点，其 children 包含用户在脑图中创建的所有子节点。

- **响应**：
  
  ```json
  {
  "node": {
    "id": 1,
    "parentId": null,
    "type": "normal",
    "title": "登录用例集",
    "priority": null,
    "sortOrder": 0,
    "version": 1,
    "children": [
      {
        "id": 2,
        "parentId": 1,
        "type": "case",
        "title": "邮箱登录",
        "priority": "P1",
        "sortOrder": 0,
        "version": 1,
        "children": [
          {
            "id": 3,
            "parentId": 2,
            "type": "precondition",
            "title": "用户已注册且处于登录页",
            "priority": null,
            "sortOrder": 0,
            "version": 1,
            "children": []
          },
          {
            "id": 4,
            "parentId": 2,
            "type": "step",
            "title": "输入邮箱和密码",
            "priority": null,
            "sortOrder": 1,
            "version": 1,
            "children": []
          },
          {
            "id": 5,
            "parentId": 2,
            "type": "expected",
            "title": "成功跳转到首页",
            "priority": null,
            "sortOrder": 2,
            "version": 1,
            "children": []
          }
        ]
      }
    ]
  },
  "layout": { "...": "..." }
  }
  ```

- **字段说明**：node 为嵌套的根节点对象，每个节点包含 id、parentId、type、title、priority、sortOrder、version 和 children 数组。根节点的 parentId 为 null，叶子节点的 children 为空数组。

#### 3.3.5 获取用例详情

- **路径**：`GET /api/project/cases/:caseId`
- **响应**：包含 type、priority、title 等完整属性。

### 3.4 测试评审管理接口

#### 3.4.1 获取评审列表

- **路径**：`GET /api/project/reviews`

- **参数**：`status`（可选，in_progress/completed）、`page`、`pageSize`

- **响应**：
  
  ```json
  {
  "records": [
    {
      "id": 1,
      "title": "登录模块评审",
      "initiator": { "id": 1, "name": "张三" },
      "status": "in_progress",
      "participantCount": 3,
      "createdAt": "2026-07-01T10:00:00Z"
    }
  ],
  "total": 10
  }
  ```

#### 3.4.2 创建评审

- **路径**：`POST /api/project/reviews`

- **请求体**：
  
  ```json
  {
  "title": "登录模块评审",
  "description": "...",
  "participantIds": [2, 3],
  "selectedNodes": [
    { "documentId": 10, "caseIds": [101, 102] }
  ]
  }
  ```

- **处理**：生成模块快照和节点快照（完整拷贝节点属性），标记 is_associated，初始化评审状态为 in_progress，发起人默认为当前用户。

- **响应**：201，返回评审基本信息。

#### 3.4.3 获取评审详情

- **路径**：`GET /api/project/reviews/:id`
- **响应**：基本信息 + participantIds + 快照信息（涉及的文档数量等）。

#### 3.4.4 获取评审快照树

- **路径**：`GET /api/project/reviews/:id/modules`
- **参数**：`documentId`（可选，按文档过滤）
- **响应**：裁剪后的模块树 + 节点树。节点部分以嵌套的根节点对象形式返回（结构与 3.3.4 一致），每个节点包含完整拷贝属性及 lastMark、associated 等快照特有字段。

#### 3.4.5 提交评审记录

- **路径**：`POST /api/project/reviews/:id/records`

- **请求体**：
  
  ```json
  {
  "snapshotNodeId": 5001,
  "operationType": "mark",
  "mark": "pass",
  "comment": "步骤清晰"
  }
  ```

- **处理**：新增 test_review_record，更新 test_review_node_snapshot 的 last_mark、last_reviewer_id、last_reviewed_at。仅 type='case' 节点可标记，所有节点均可评论。

#### 3.4.6 获取节点评审记录

- **路径**：`GET /api/project/reviews/:id/nodes/:nodeId/records`
- **响应**：该节点的所有标记和评论历史，按时间排序。

#### 3.4.7 完成评审

- **处理**：根据 original_node_id 匹配当前节点，更新快照属性；已删除节点标记 is_deleted=1；保留评审记录不变。

## 4. 业务逻辑设计

### 4.1 文档创建与默认根节点

创建 type=document 的模块节点时，事务中插入 test_case_module 记录，并在 test_case_node 表中创建默认根节点（title 与文档名称相同，type='normal'，sort_order=0，parent_id 为 NULL）。此根节点作为文档脑图的唯一顶层节点，用户可在其下添加子节点并标记为不同用例类型。获取文档节点树时，该根节点作为顶层节点返回，其 children 包含所有用户创建的节点。

### 4.2 脑图实时协作

通过 WebSocket 连接 `/ws/documents/:docId`，客户端加入文档房间。操作消息包含 op（add_node、move_node、update_attrs、delete_node、update_layout）及 payload。服务端校验权限后更新 test_case_node 和 test_case_document_layout，广播给同房间其他用户。冲突处理使用节点 version 字段的乐观锁机制。脑图数据以嵌套树形结构在前端组件中渲染，保存时递归遍历节点树并更新数据库。

### 4.3 快照生成与裁剪

评审或计划创建时，递归复制所需路径的 test_case_module 和涉及文档的全部 test_case_node。节点所有字段原样复制到快照表，is_associated 根据用户选择的 caseIds 标记。裁剪展示时根据关联节点向上追溯祖先、向下加载后代，构建最小树，返回嵌套的根节点对象。

### 4.4 评审/执行记录与状态更新

每次提交记录时，事务中插入记录表并更新快照节点的最后状态字段。评审中仅用例节点可标记，所有节点可评论。执行中仅关联用例节点可标记执行结果，默认根节点不可执行。

### 4.5 同步最新用例

触发后批量对比原始节点与当前节点，更新快照属性；若原始节点不存在，快照节点 is_deleted=1。评审记录/执行记录不受影响，最终状态保留。

### 4.6 权限控制

所有接口校验用户是否属于当前项目所属的工作空间。评审操作仅发起人可完成/同步，参与者均可提交记录。计划操作负责人可编辑、同步、关闭，执行人可提交执行记录。

详情页复用脑图组件（mode='plan'），快照树数据以嵌套根节点对象形式返回，工具栏切换为执行操作按钮，节点显示执行状态图标。

**交互说明**：

| 操作     | 触发方式                 | 反馈                 |
| ------ | -------------------- | ------------------ |
| 标记执行结果 | 点击关联用例节点 → 选择结果 + 备注 | 提交执行记录，节点颜色更新      |
| 查看执行历史 | 点击节点查看详情             | 显示执行记录列表           |
| 关闭计划   | 负责人点击[关闭计划]          | 检查未执行用例 → 确认后关闭    |
| 同步用例   | 负责人点击[同步]            | 确认 → 更新快照属性，保留执行记录 |
