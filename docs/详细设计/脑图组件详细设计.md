## 1. 概述

脑图组件是功能测试模块的核心组件，用于测试用例的可视化编辑、评审与执行跟踪。采用 **kityminder-core** 作为脑图渲染引擎，通过 **Yjs** 实现多人实时协作。组件支持三种模式，通过 `mode` 属性切换：

- **编辑模式（edit）**：完整的节点增删改、类型标记、优先级标记，数据实时同步至 `test_case_node` 表。
- **评审模式（review）**：基于快照只读脑图，提供评审标记（通过/不通过/待评审）和评论功能，数据写入 `test_review_record` 表并更新快照节点状态。
- **计划模式（plan）**：基于快照只读脑图，提供执行结果标记（通过/失败/阻塞/未执行），数据写入 `test_plan_execution_record` 表并更新快照节点状态。

## 2. 技术选型

| 组件    | 技术                                            | 说明                            |
| ----- | --------------------------------------------- | ----------------------------- |
| 脑图核心  | kityminder-core                               | 基于 SVG 的脑图编辑器，支持节点增删改、拖拽、主题定制 |
| 数据绑定  | kityminder-core 的 `importData` / `exportData` | JSON 格式数据中间层                  |
| 实时协作  | Yjs (y-websocket)                             | 编辑模式下增量同步，评审/计划模式不使用          |
| 前端框架  | Vue 3 + Composition API                       | 封装为 Vue 组件，管理生命周期和交互          |
| UI 增强 | Element Plus (弹出菜单、工具栏、评论抽屉)                  | 节点右键菜单、悬浮工具栏、评论抽屉             |

## 3. 组件架构

```
Vue组件：MindMapEditor.vue

```

## 4. 数据模型与模式映射

### 4.1 节点类型定义

| 类型值            | 含义          | 图标/样式         |
| -------------- | ----------- | ------------- |
| `normal`       | 普通节点（无特殊标记） | 默认样式          |
| `precondition` | 前置条件        | 蓝色标记          |
| `step`         | 执行步骤        | 绿色标记          |
| `expected`     | 预期结果        | 橙色标记          |
| `case`         | 测试用例        | 用例图标，可显示优先级色标 |

### 4.2 数据来源与节点结构

文档的脑图数据以嵌套的根节点对象形式从后端获取。根节点是创建文档时自动生成的默认节点，其 `children` 数组包含用户在脑图中创建的所有子节点，每个子节点可继续嵌套。

**后端接口响应格式**（`GET /api/project/documents/:docId/nodes`）：

```json
{
  "node": {
    "id": 1,
    "parentId": null,
    "type": "normal",
    "title": "登录用例集",
    "priority": null,
    "sortOrder": 0,
    "version": 1,
    "children": [
      {
        "id": 2,
        "parentId": 1,
        "type": "case",
        "title": "邮箱登录",
        "priority": "P1",
        "sortOrder": 0,
        "version": 1,
        "children": [
          {
            "id": 3,
            "parentId": 2,
            "type": "precondition",
            "title": "用户已注册且处于登录页",
            "priority": null,
            "sortOrder": 0,
            "version": 1,
            "children": []
          },
          {
            "id": 4,
            "parentId": 2,
            "type": "step",
            "title": "输入邮箱和密码",
            "priority": null,
            "sortOrder": 1,
            "version": 1,
            "children": []
          },
          {
            "id": 5,
            "parentId": 2,
            "type": "expected",
            "title": "成功跳转到首页",
            "priority": null,
            "sortOrder": 2,
            "version": 1,
            "children": []
          }
        ]
      }
    ]
  },
  "layout": { "...": "..." }
}
```

评审快照树（`GET /api/project/reviews/:id/tree`）和计划快照树（`GET /api/project/plans/:id/tree`）的节点部分采用相同结构，额外包含 `reviewStatus`、`executionStatus`、`relatedBugIds`、`comments` 等快照特有字段。

### 4.3 kityminder 节点数据结构（统一模型）

```json
{
  "data": {
    "id": "node-uuid",
    "text": "登录用例集",
    "type": "case",
    "priority": "P2",
    "reviewStatus": { "result": "pass", "reviewer": "张三", "time": "2026-07-06T10:30:00Z" },
    "executionStatus": { "result": "pass", "executor": "李四", "time": "2026-07-05T14:20:00Z" },
    "relatedBugIds": [101, 102],
    "comments": [
      { "author": "王五", "content": "步骤需要补充", "time": "2026-07-04T09:15:00Z" }
    ]
  },
  "children": [...]
}
```

### 4.4 三种模式的数据来源与操作目标

| 模式         | 数据来源                                                  | 可编辑字段                             | 操作目标表                                                                        |
| ---------- | ----------------------------------------------------- | --------------------------------- | ---------------------------------------------------------------------------- |
| **edit**   | `GET /api/project/documents/:docId/nodes` 返回的嵌套根节点与布局 | 全部字段（type, priority, text 等）      | `test_case_node`（通过 WebSocket 异步持久化）                                         |
| **review** | `GET /api/project/reviews/:id/tree` 返回的快照节点树          | `reviewStatus`（标记）、`comments`（评论） | `test_review_node_snapshot`（更新 last_mark），`test_review_record`（新增记录）         |
| **plan**   | `GET /api/project/plans/:id/tree` 返回的快照节点树            | `executionStatus`（执行结果）           | `test_plan_node_snapshot`（更新 last_result），`test_plan_execution_record`（新增记录） |

在评审/计划模式下，`type`、`priority`、`text` 等原始用例字段为只读，来自快照表，不可通过脑图修改。评审/计划模式下的状态字段由后端在加载快照树时填充，前端提交标记后即时更新。

## 5. 工具栏设计

工具栏悬浮于脑图画布顶部，水平居中，背景半透明磨砂。根据 `mode` 属性显示不同按钮组。

### 5.1 编辑模式（mode='edit'）

| 按钮                | 功能                | 快捷键             | 说明      |
| ----------------- | ----------------- | --------------- | ------- |
| 标记为测试用例           | 标记为测试用例，无优先级时默认P2 | —               |         |
| 标记为前置条件           | 标记为前置条件           | —               |         |
| 标记为执行步骤           | 标记为执行步骤           | —               |         |
| 标记为预期结果           | 标记为预期结果           | —               |         |
| P0 / P1 / P2 / P3 | 设置优先级并自动标记为测试用例   | —               |         |
| ──────────        |                   |                 |         |
| 添加子节点             | 新增普通子节点           | Tab             |         |
| 添加兄弟节点            | 新增普通兄弟节点          | Enter           |         |
| 删除节点              | 删除选中节点            | Delete          |         |
| ──────────        |                   |                 |         |
| 撤销 / 重做           |                   | Ctrl+Z / Ctrl+Y |         |
| 放大 / 缩小 / 适应      | 缩放控制              | Ctrl+= / - / 0  |         |
| 切换抓手状态            | 切换画布拖拽模式          | H               | 高亮显示激活态 |

**标记联动规则**：

- **标记为测试用例**：`type` 设为 `case`；若 `priority` 为空，自动设为 `"P2"`。
- **标记优先级（P0-P3）**：设置 `priority` 为所选值；若 `type` 不为 `case`，自动改为 `case`。
- **标记为前置条件/步骤/预期结果**：`type` 设为对应值；`priority` 保持不变。

### 5.2 评审模式（mode='review'）

| 按钮         | 功能                                               | 说明                   |
| ---------- | ------------------------------------------------ | -------------------- |
| ✅ 通过       | 标记通过，提交至 `POST /api/project/reviews/:id/records` | 仅 case 节点可用，高亮当前评审结果 |

| 💬 评论      | 打开评论抽屉，查看/添加评论，提交至同一接口                           | 所有节点可用               |

### 5.3 计划模式（mode='plan'）

| 按钮         | 功能                                             | 说明                   |
| ---------- | ---------------------------------------------- | -------------------- |
| ✅ 通过       | 标记通过，提交至 `POST /api/project/plans/:id/records` | 仅 case 节点可用，高亮当前执行结果 |
| ❌ 失败       | 标记失败，同上                                        |                      |

## 8. 关联 Bug 链接标签

- **显示条件**：节点 `data.relatedBugIds` 不为空，且至少含有一个有效 Bug ID。
- **渲染位置**：在节点文本下方，水平排列多个可点击的链接标签。
- **标签样式**：小型标签，浅红色背景，文字为 `#101` 形式，悬停时背景加深并显示手型光标。
- **点击行为**：调用 `window.open('/workspace/projects/bugs/' + bugId, '_blank')` 在新标签页打开 Bug 详情页。点击事件必须调用 `event.stopPropagation()` 防止触发节点选中。
- **数据来源**：编辑模式从 `test_case_node` 表关联查询获得；评审/计划模式从快照表的冗余字段获得。
- **实现方式**：扩展 kityminder 的节点渲染模板，使用 SVG `foreignObject` 嵌入 HTML 标签，并绑定点击事件。

## 9. 评论功能（评审模式）

### 9.1 评论抽屉

- 打开方式：工具栏“💬评论”按钮或右键“添加评论”。
- 展示当前节点所有评论，按时间排序，显示作者、时间、内容。
- 底部输入框 + 发送按钮，支持 Enter 快捷发送。
- 提交后通过 HTTP 请求新增评审记录（调用 `POST /api/project/reviews/:id/records`，operationType='comment'），并更新本地节点 `comments` 数组。

### 9.2 评论同步

多用户同时评审时，评论通过提交后刷新本地数据实现同步；后续可扩展为 Yjs 子文档协作。

## 10. 初始化加载流程

### 10.1 编辑模式加载

1. 用户点击模块树中的文档节点。
2. 前端调用 `GET /api/project/documents/:docId/nodes`，获取嵌套的根节点对象和布局 JSON。
3. 前端将嵌套的根节点对象转换为 kityminder 所需的 JSON 格式（根节点作为画布中心节点，其 children 递归展开）。
4. 使用 `minder.importData(json)` 渲染脑图，应用布局 JSON 恢复展开/折叠状态。
5. 建立 WebSocket 连接，加入文档房间。

### 10.2 评审模式加载

1. 用户进入评审详情页。
2. 前端调用 `GET /api/project/reviews/:id/tree`，获取裁剪后的快照树（嵌套根节点对象格式）。
3. 转换为 kityminder JSON 渲染只读脑图。
4. 工具栏切换为评审操作按钮。
5. 不建立 WebSocket 连接，标记和评论通过 HTTP 提交。

### 10.3 计划模式加载

1. 用户进入计划详情页。
2. 前端调用 `GET /api/project/plans/:id/tree`，获取裁剪后的快照树（嵌套根节点对象格式）。
3. 转换为 kityminder JSON 渲染只读脑图。
4. 工具栏切换为执行操作按钮。
5. 不建立 WebSocket 连接，执行结果通过 HTTP 提交。

## 11. 实时协作设计（基于 Yjs）

### 11.1 协作数据模型

- 使用单个 Y.Doc 存储文档的脑图 JSON。
- `ydoc.getMap('mindmap')` 存储完整的嵌套 JSON 对象（直接使用 kityminder 的 export JSON）。
- 每当 kityminder 内容变化（`contentchange`），将最新的 JSON `exportData()` 设置到 Y.Map 中，Yjs 自动计算差异并同步。
- 从 Y.Map 的 `observe` 事件接收远程变更，调用 `minder.importData()` 更新画布，同时保留本地光标状态。

### 11.2 冲突处理

- Yjs 使用 CRDT 算法自动解决 JSON 并发修改冲突。
- 文档节点数较小时（<500）采用全量 JSON 替换；若节点数更多，可优化为按节点 ID 映射到 Y.Map 的嵌套结构。

### 11.3 协作感知

- 通过 `y-websocket` 连接到 `/ws/documents/{docId}`，服务端广播 Yjs 更新。
- 其他用户的光标/选中节点通过 Yjs Awareness 协议共享，在脑图上渲染彩色光标。
- 在线用户头像显示在悬浮工具栏右侧。

### 11.4 编辑模式保存

编辑模式下不通过 HTTP 接口手动保存，所有变更通过 WebSocket 实时同步。后端 WebSocket 处理器接收到 Yjs 更新后，解析嵌套的文档 JSON，递归遍历节点树，执行与 `test_case_node` 表的差异化更新（新增、修改、删除节点），并更新 `test_case_document_layout` 表。

## 12. 与后端交互总结

| 场景        | 模式     | 接口                                        |
| --------- | ------ | ----------------------------------------- |
| 加载编辑数据    | edit   | `GET /api/project/documents/:docId/nodes` |
| 实时保存      | edit   | WebSocket `/ws/documents/{docId}`         |
| 加载评审快照树   | review | `GET /api/project/reviews/:id/tree`       |
| 提交评审标记/评论 | review | `POST /api/project/reviews/:id/records`   |
| 加载计划快照树   | plan   | `GET /api/project/plans/:id/tree`         |
| 提交执行结果    | plan   | `POST /api/project/plans/:id/records`     |

## 14. 实施要点

1. **模式切换**：父组件传入 `mode` 和对应 ID（文档ID/评审ID/计划ID），组件根据 mode 加载不同数据源。
2. **数据加载**：编辑模式调用文档节点接口获取嵌套根节点对象；评审/计划模式调用快照树接口获取相同结构的数据。前端将嵌套对象转换为 kityminder JSON 后渲染。
3. **数据隔离**：编辑模式与快照模式使用不同的 Yjs 房间或禁用协作（快照模式下无需实时同步，标记通过 HTTP 提交）。
4. **权限控制**：评审模式仅参与者可标记/评论；计划模式仅负责人/执行人可标记结果。前端通过传入的 `editable` 标记控制，后端接口校验。
5. **kityminder 扩展**：自定义节点渲染模板，显示 Bug 链接标签、评审/执行图标；使用 `foreignObject` 嵌入 HTML 元素。
6. **工具栏状态响应**：监听 kityminder 的 `selectionchange` 事件，更新按钮高亮。
7. **右键菜单**：使用 Element Plus `Dropdown` 组件，根据当前模式动态渲染菜单项。
8. **抓手模式**：`toggleGrab` 切换 `isGrabMode`，通过设置 kityminder 的只读+拖拽模式实现平移。
9. **WebSocket 持久化**：后端接收到 Yjs 更新后，递归解析嵌套的文档 JSON 树，比对数据库现有节点，执行新增、更新、删除操作，并更新布局表。
10. **Bug 链接跳转**：点击关联 Bug 标签时，以新窗口打开缺陷详情页，路径为 `/workspace/projects/bugs/:bugId`。
